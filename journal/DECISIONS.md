- **D-001 · 2026-07-16 · Architecture = modular monolith ("modulith") + a few satellites, one versioned appliance.** Rejected full microservices — a 4–8 dev team can't operate them across customer VMs; the adversarial design review repeatedly pruned toward this. [spec §2]
- **D-002 · 2026-07-16 · Backend = TypeScript/NestJS core + Python FastAPI ML sidecars.** One language for ~95% of the app + end-to-end typed contracts; Python only where ML libraries genuinely require it. [spec §1, §4]

Multi-provider (Claude / OpenAI / local) chosen per customer via the `DATA_EGRESS` gate. The five AI-safety invariants are law. [spec §3]
- **D-004 · 2026-07-16 · Data + async = PostgreSQL 18 + pgvector + Drizzle + transactional outbox + pg-boss. No Restate/Temporal/Kafka/Redis at v1.** One durability model, one backup story; long-running/human-gated flows are Postgres-backed sagas. [spec §4, §9.2]
- **D-005 · 2026-07-16 · Realtime = Socket.IO in-process, outbox reconcile-on-connect; Valkey (cache + pub/sub) only when multi-node.** [spec §4, §9.5]

- **D-007 · 2026-07-16 · Frontend = static-export marketing + Vite React SPA; ECharts (one lib); Money = NUMERIC + decimal.js (integer minor-units only for the ledger).** Award-caliber 3D landing scope decided at the marketing-page phase. [spec §6, §9.5]
- **D-008 · 2026-07-16 · Delivery = prebuilt Hyper-V/VMware Linux VM appliance; plain Docker Compose + an admin-triggered appliance-agent doing cosign-signed blue/green updates over the customer VPN.** Rejected Docker Swarm / k3s on a single box. [spec §9.8]

Testcontainers is unaffected — it binds no host volume. [Phase 1 Task 6]
- **D-017 · 2026-07-17 · `@erp/db` adds `@types/pg` (catalog `^8.11.0`; `pg` ships no types) and points `drizzle.config.ts` `schema` at the concrete `src/schema/platform.ts`, not the `index.ts` barrel.** drizzle-kit loads the config via esbuild-register (CJS), which can't resolve the barrel's NodeNext `.js`→`.ts` re-export; runtime code still imports via the barrel. Nest DI under Vitest needs `reflect-metadata` loaded before modules import (added to `setupFiles`). `apps/api` stays CommonJS; the ESM/CJS seam with the all-ESM `@erp/*` packages remains deferred to Phase 2 (see D-014, and the plan's Task 8 `/ready` follow-up). - **D-022 · 2026-07-21 · PG schemas named by eventual owner (`platform`/`md`/`wh`/`fin`); RLS = policies TO public + ENABLE+FORCE on every tenant table + `SET LOCAL ROLE app_rw` inside `withTenantTx`; superuser is the deliberate bootstrap/migration/relay/seed bypass.** Fail-closed context = `NULLIF(current_setting('app.tenant_id', true), '')::uuid`; the catalog test asserts ENABLE+FORCE via a fail-closed **blocklist** over all non-system schemas (covers future tables automatically). **D-017 extended:** drizzle-kit's config loader cannot resolve ANY cross-file imports between schema files — every schema file inlines its `pgSchema(...)` + tenant-isolation policy (with keep-in-sync comments), and FKs that would cross files ship as plain SQL in custom migrations. `CREATE ROLE app_rw` is guarded idempotent (roles are cluster-wide, migration state per-database). [Phase 2 Task 3; spec §7, §9.4]
- **D-023 · 2026-07-21 · Audit chains serialize per aggregate via a row-locked `audit_head` upsert; the log stores payload hashes only (PII-free by construction).** Hardening from execution-time review: `verifyAuditChain` reads log + head in one REPEATABLE READ snapshot (no phantom tamper alarms on hot aggregates); wiped logs detected via head-seq vs log-length; `tenantId`/`actor` validated + normalized to canonical lowercase UUID before hashing (write/verify symmetry); `stableStringify` honors `toJSON` and JSON semantics for `undefined`. Rule: a transaction appending audits for multiple aggregates must append in canonical (aggregateType, aggregateId) order — ABBA deadlock prevention. UPDATE/DELETE/TRUNCATE on `audit_log` blocked by trigger even for the owner. [Phase 2 Task 4; spec §7]
- **D-024 · 2026-07-21 · Outbox relay = reentrancy-guarded interval loop in `apps/worker` (pg-boss cron floor is 1 min), at-least-once with `singletonKey` best-effort dedupe; consumer idempotency is the real guarantee.** Jobs carry the camelCase §8 DomainEvent envelope (the Phase-3 wire contract); consumers must not assume the archive row is visible when a job arrives. The relay/chain-verify connection must be **superuser or BYPASSRLS** (under FORCE RLS a mere table owner silently sees zero rows) — provisioning-runbook item for Phase 3. Poison-row quarantine + attempt cap MUST land with the first real consumer registration (a deterministically failing row currently wedges the batch; safe now — the prod registry is empty). [Phase 2 Task 5; spec §8, §9.2]
- **D-025 · 2026-07-21 · Gapless ranges: row-locked `UPDATE … RETURNING` + in-tx allocations journal; gaps are DETECTED and explainable, not prevented.** Journal uniqueness `(tenant, range, period, value)` is the DB-level double-allocation tripwire; `detectGaps` finds journal holes (= journal/counter divergence; a lost document leaves a dangling `docRef`, which needs a future dangling-docRef check, not detectGaps). Allocate-late convention documented, incl. the shared-audit-aggregate lock-order rule. `allocateNumber`'s journal row IS its audit record — the enclosing business transaction audits the document that consumed the number (deliberate scoped exception to "every mutation writes an audit row": chain-auditing each allocation would add a second serialization point for no extra information). Every Phase-2 range is treated legally-gapless; a merely-unique series mode arrives with its first consumer. Fiscal corollary: the period gate (`assertPeriodOpen`) takes a `FOR SHARE` row lock so an in-flight close and an in-flight posting serialize instead of racing. [Phase 2 Tasks 8–9; spec §6]
