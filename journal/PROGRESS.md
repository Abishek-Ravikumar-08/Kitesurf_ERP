### 2026-07-21 — PR #2 first CI run failed: latent pool-error crash found & fixed
- The `pull_request` CI run failed with **all 62 int tests passing** — Vitest counted 1 unhandled error: FATAL 57P01 (`terminating connection due to administrator command`) from an idle pooled client during Testcontainers teardown. Root cause (systematic-debugging, reproduced deterministically): **`makeDb` attached no `pool.on("error")` listener** — node-postgres pools re-emit idle-client errors, and an unlistened `'error'` event throws as an uncaught exception. Windows-local timing never hit the race; the Linux runner did. This was a latent **production** bug: an appliance Postgres restart (blue/green update) would have crashed the api/worker the same way.
- Fix: pool-level error handler in `makeDb` (log; pool discards the dead client and self-heals; `/ready`/boot gates report unavailability meanwhile) + a deterministic regression test (`pool-error.int.test.ts`: `pg_terminate_backend` on an idle pooled client → handled, pool self-heals) that reproduced the exact CI failure signature RED before the fix. db int now 13/13; platform 44/44 with zero unhandled errors. Pushed to the PR branch — confirm CI green on the re-run.
